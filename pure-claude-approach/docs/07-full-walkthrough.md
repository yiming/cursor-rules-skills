# Full Walkthrough：從頭跑一遍所有功能

本指引帶你依序操作系統的所有功能，從簡單到複雜，串起一條完整的故事線。

> **前提**：使用 Claude Code，工作目錄設為 `pure-claude-approach/`。

### Tips：兩種觸發方式

每個步驟都可以用兩種方式操作：

```
# 方式 A：指令 + ID（精確觸發）
/process-order BR250420-001

# 方式 B：自然語言 + @檔案引用（意圖觸發）
請處理這張訂單 @orders/BR250420-001.req.md
```

方式 B 的好處是 `@` 會帶出檔案選擇 UI，不用記 ID 也不用手打路徑。
兩者結果相同——方式 A 走 Skill 的 `$ARGUMENTS` 參數，方式 B 靠 Claude 從 CLAUDE.md 團隊分工表找到對應 Skill。

以下步驟以方式 A 為主，你也可以隨時切換成方式 B。

---

## 故事背景

UU 大飯店接到以下一系列需求：

1. 林小芳訂房 2 晚（已有需求檔）
2. 林小芳加訂下午茶（已有需求檔）
3. 張大偉預約婚宴場地（120 人，已有需求檔 + 名單）
4. 黃威翔預約企業會議（35 人，無名單）
5. 陳美玲預約生日派對（18 人，名單內嵌）

按順序操作，你會體驗到：訂房 → 訂餐 → 場地（三種 guest_list 模式）→ 桌牌 prompt 串接。

## Step 1：查空房

先確認 4/20-4/22 有沒有房間可用。

```
/check-availability 2025-04-20 2025-04-22
```

**觀察重點**：
- Skill 透過 `` !`command` `` 動態注入房間清單與已訂房資料
- 輸出會顯示各房型的可用房號

---

## Step 2：處理訂房（簡單）

處理林小芳的訂房需求。

```
/process-order BR250420-001
```

**預期結果**：
- 產出 `orders/BR250420-001-確認單.md` + `orders/BR250420-001-任務.md`
- 房型：STD-D（標準雙人房），3200 × 2 晚 = 6,400 元
- auditor 自動驗證金額與房號

**觀察重點**：
- order-clerk（sonnet）處理 → auditor（haiku）驗證，兩個角色、兩個模型
- 所有金額可追溯到 `master-data/room-types.yaml`

---

## Step 3：處理訂餐（中等）

林小芳加訂下午茶，關聯到她的訂房單。

```
/process-dining-order DR250421-001
```

**預期結果**：
- 英式下午茶 ×1（800）+ 拿鐵 ×2（150×2）= 1,100 元
- 低消檢查：1,100 ÷ 2 = 550 元/人 → ✓ 皆通過（門檻 300 元）
- 預算 1,200 元 → ✓ 在預算內
- 確認單中顯示關聯訂房 BR250420-001

**觀察重點**：
- 低消檢查是本 Skill 的核心邏輯
- `related_order` 跨訂單關聯

---

## Step 4：處理場地 — 婚宴（複雜，外部名單）

張大偉的 120 人婚宴，名單為外部 YAML 檔案。

```
/process-venue-order VR250420-001
```

**預期結果**：

| 項目 | 內容 |
|------|------|
| 場地 | VN-GRAND 大宴會廳（宴會容量 150 >= 120） |
| 場地費 | 30,000 元（晚宴場） |
| 餐標 | BQ-B 豪華宴會餐 1,800 × 120 = 216,000 元 |
| 花藝 | 8,000 元 |
| 攝影 | 12,000 元 |
| LED 背板 | 5,000 元 |
| **合計** | **271,000 元** |
| 預算 | 250,000 元 → ⚠ 超出 21,000 元 |

桌次安排（13 桌）：
- 第 1 桌：主桌 8 人（新郎新娘 + 雙方父母 + 證婚人 + 主婚人）
- 第 2-3 桌：男方親友（20 人）
- 第 4-5 桌：女方親友（20 人）
- 第 6-7 桌：同事（20 人）
- 第 8-13 桌：朋友（32 人）
- 素食 3 位分散標記

**觀察重點**：
- `guest_list: VR250420-001-名單.yaml` → 外部檔案模式
- 場地 + 餐飲混合計價
- 桌次安排演算法（同組同桌、主桌優先）
- 確認單末尾提示可執行 `/generate-card-prompt`

## Step 5：產生桌牌 Prompt → 交外部產圖（多 Skill 串接）

這一步展示 **AI-to-AI 接力**：Claude 產出 prompt 檔，再交給另一個 AI（Cursor + Gemini）產生圖片。

### Step 5a：Claude 產出 prompt 檔

```
/generate-card-prompt VR250420-001
```

**產出**：`orders/VR250420-001-card-prompt.md`

這個檔案是一份**給圖像生成 AI 的工作指令**，內容包含：
- 角色設定（「你是宴會視覺設計 Agent」）
- 風格規範（中國水墨風：宣紙質感、書法水墨點綴…）
- 待處理名單（從確認單桌次表提取的桌名 + 賓客）
- 產出路徑規則（圖片存到哪裡、檔名怎麼命名）

因為 120 人 > 50 人門檻，prompt 指定產出**桌牌**（每桌一張，共約 13 張），不逐人產座位卡。

### Step 5b：交給 Cursor + Gemini 產圖

**產生主桌VIP個人座位卡**:
打開 Cursor（或任何有圖像生成能力的工具），建議使用模型: Gemini 3.1 Pro (或更新版本)
把 prompt 檔的內容貼入：

```
請依照 @orders/VR250420-001-card-prompt.md 的指示，批次生成桌牌圖片
```

**產生主桌VIP個人座位卡**:
```
請依照 @pure-claude-approach/orders/VR250420-001-card-prompt.md 的名單，
加上 mode: 座位卡 指令，幫我生成主桌 8 位 VIP 的個人座位卡圖片。
```

**預期結果**：
- 在 `orders/VR250420-001-cards/` 資料夾下產出圖片
- 檔名如 `card-主桌.png`、`card-男方親友A.png`、`card-男方親友B.png`…
- 每張圖片為中國水墨風格的桌牌

> **為什麼分兩步？** Claude Code 本身沒有圖像生成能力。它負責「想好要做什麼」（prompt），實際「畫出來」由 Gemini 等圖像模型執行。這就是 AI-to-AI 接力的意義：各自做擅長的事。

**觀察重點**：
- **多 Skill 串接**：process-venue-order 的確認單 → generate-card-prompt 讀取桌次表 → 產出 prompt
- **跨工具協作**：Claude Code 產 prompt.md → Cursor + Gemini 產圖片
- print-designer agent 負責執行 generate-card-prompt skill

## Step 6：處理場地 — 企業會議（無名單）

黃威翔的 35 人會議，不需要名單也不需要桌牌。

```
/process-venue-order VR250421-001
```

**預期結果**：

| 項目 | 內容 |
|------|------|
| 場地 | VN-ROSE 玫瑰廳（教室容量 40 >= 35） |
| 場地費 | 15,000 × 2.5（全日場）= 37,500 元 |
| 餐標 | BQ-C 西式自助餐 800 × 35 = 28,000 元 |
| 額外投影機 | 2,000 × 1 = 2,000 元 |
| VIP 停車 | 500 × 5 = 2,500 元 |
| **合計** | **70,000 元** |
| 預算 | 80,000 元 → ✓ 在預算內 |

**觀察重點**：
- `guest_list` 未填 → 無名單模式，不執行桌次安排
- 全日場 price_multiplier: 2.5 的計價邏輯
- 教室式 → 使用 `capacity_classroom` 判斷容量

## Step 7：處理場地 — 生日派對（內嵌名單）

陳美玲的 18 人生日派對，名單直接寫在 req.md 內。

```
/process-venue-order VR250422-001
```

**預期結果**：

| 項目 | 內容 |
|------|------|
| 場地 | VN-SKY 星空露臺（宴會容量 40 >= 18） |
| 場地費 | 20,000 元（下午場） |
| 餐標 | BQ-D 下午茶派對 500 × 18 = 9,000 元 |
| 花藝 | 8,000 元 |
| **合計** | **37,000 元** |
| 預算 | 25,000 元 → ⚠ 超出 12,000 元 |

桌次安排（2 桌）：
- 第 1 桌：家人 6 人 + 閨蜜 4 人 = 10 人
- 第 2 桌：閨蜜 2 人 + 同事 6 人 = 8 人
- 素食 1 位（李美珍）個別標記

**觀察重點**：
- `guest_list: embedded` → 名單從 req.md 內文的 YAML 區塊解析
- 小型活動，2 桌就夠
- 超出預算 → 確認單應建議替代方案（如改餐標或減加購）

---

## Step 8：生日派對桌牌（花園自然風，逐人座位卡）

```
/generate-card-prompt VR250422-001
```

**產出**：`orders/VR250422-001-card-prompt.md`

與 Step 5 的差異：

| | Step 5（婚宴） | Step 8（生日派對） |
|---|---|---|
| 人數 | 120 人（> 50） | 18 人（<= 50） |
| 產出模式 | 桌牌（每桌一張，~13 張） | 座位卡（每人一張，18 張） |
| 風格 | 中國水墨風 | 花園自然風 |
| 圖片路徑 | `orders/VR250420-001-cards/` | `orders/VR250422-001-cards/` |
| 檔名 | `card-主桌.png` | `card-陳美玲.png` |

同樣需要將 prompt 檔交給 Cursor + Gemini 產出實際圖片（參考 Step 5b）。

## 進階挑戰：雙場地婚宴

如果你想測試最複雜的邊界案例：

```
/process-venue-order VR250425-001
```

這張訂單是**跨場地**：星空露臺（下午證婚）+ 玫瑰廳（晚宴），考驗 Skill 能否處理：
- 雙場地費用加總
- 花藝 ×2（兩場地各一套）
- 餐標只算晚宴（露臺證婚不供餐）
- 55 人桌次安排（6 桌）
