# 特性分析：Claude Code 獨有機制

## 從「規則遵循」到「團隊協作」

舊版展示的是：Claude 能讀規則、按步驟做事。
新版展示的是：Claude 能**組建團隊、委派任務、分工驗證**。

這不是功能升級——是哲學轉變。

## 九大 Claude 獨有機制

### 1. 指令式 Skill

**Cursor**：Skill 被 Rule 的 glob 被動觸發，使用者不知道什麼時候會啟動。
**Claude**：`/process-order BR250315-001` 就像 `/commit` 一樣，是一個使用者主動呼叫的指令。

```
# Cursor: 被動
使用者打開 orders/BR250315-001.req.md → glob 匹配 → Rule 注入 → Skill 被動啟動

# Claude: 主動
使用者輸入 /process-order BR250315-001 → Skill 直接執行
```

### 2. 動態上下文注入

Cursor 的 SKILL.md 是純靜態文件。Claude 的 SKILL.md 可以在載入時執行 shell 指令：

```markdown
## 現有房間
!`cat master-data/room-list.yaml`

## 已被預訂的房間
!`grep -r "建議房號" orders/*-確認單.md 2>/dev/null`
```

Skill 載入時就已經拿到即時資料，不需要額外指示 AI 去查。

### 3. 子代理委派

**Cursor**：只有一個 AI，讀一組規則，做所有事。
**Claude**：主 Claude 可以委派給不同的子代理：

- **order-clerk**（sonnet）：處理訂單，有讀寫權限
- **auditor**（haiku）：驗證結果，只有讀取權限

每個子代理在自己的 context window 裡獨立工作。

### 4. 角色權限分離

每個子代理有不同的工具集：

| 子代理 | 可用工具 | 不可用工具 |
|--------|---------|-----------|
| order-clerk | Read, Write, Glob, Grep | 無 Edit 限制 |
| auditor | Read, Grep, Glob | Write, Edit |

這確保了「處理」和「驗證」由不同角色完成，互不干擾。

### 5. 模型分配

- **order-clerk** 用 sonnet：需要理解複雜需求、計算匹配
- **auditor** 用 haiku：只需比對數字，快且便宜

Claude Code 可以為不同任務選擇最合適的模型。

### 6. 信任邊界

**Cursor**：`.mdc` 寫「不得修改 *.req.md」——但這只是建議，AI 可以無視。
**Claude**：`settings.json` 的 `deny` 規則是系統層級的強制保護：

```json
{
  "permissions": {
    "deny": ["Edit(orders/*.req.md)", "Edit(master-data/*)"]
  }
}
```

AI 想改也改不了。這是「指導」和「強制」的本質差異。

### 7. 目錄級規則繼承

每個子目錄的 `CLAUDE.md` 在操作該目錄時自動載入：

```
CLAUDE.md                    ← 全域政策（永遠生效）
├── master-data/CLAUDE.md    ← 主資料規則（操作 master-data/ 時生效）
└── orders/CLAUDE.md         ← 訂單規則（操作 orders/ 時生效）
```

不需要 glob 模式匹配，目錄位置本身就是作用域。

### 8. 模板分離

**Cursor**：模板寫在 SKILL.md 裡面，和流程邏輯混在一起。
**Claude**：模板獨立存放在 `templates/` 子目錄：

```
.claude/skills/process-order/
├── SKILL.md              ← 流程邏輯
└── templates/
    ├── confirmation.md   ← 確認單模板
    └── task-list.md      ← 任務清單模板
```

修改模板不影響流程，修改流程不影響模板。

### 9. 參數化 Skill

Skill 支援 `$ARGUMENTS`、`$1`、`$2` 等變數替換：

```
/process-order BR250315-001    → $ARGUMENTS = "BR250315-001"
/check-availability 2025-04-20 2025-04-22  → $1 = "2025-04-20", $2 = "2025-04-22"
```

Cursor 的 Skill 沒有參數概念。

## 與 Cursor 機制的完整對比

| 面向 | Cursor | Claude |
|------|--------|--------|
| 觸發機制 | glob 被動匹配 | `/指令` 主動觸發 |
| 上下文 | 靜態文件 | 動態注入（`` !`command` ``） |
| 分工 | 單一 AI | 多子代理（不同角色、模型） |
| 權限控制 | 文字建議 | settings.json 強制 |
| 規則作用域 | glob 模式 | 目錄繼承 |
| 模板管理 | 嵌在 SKILL.md 內 | 獨立 templates/ 目錄 |
| 參數 | 無 | $ARGUMENTS, $1, $2 |
| 模型選擇 | 固定 | 按角色分配 |
| 驗證 | 無內建機制 | auditor 子代理 |
